## What Site Settings control

Site Settings control how Campium behaves for your families, staff, and finance team across modules, payments, the mobile app, and admin behaviors. Each setting maps to a stored key that the platform reads whenever it renders portals, processes payments, or loads the mobile app.

Settings on this screen save automatically as you change them, so updates apply quickly without a single global Save button.

<Callout kind="info">

Site Settings are shared across your entire account. Before changing anything, confirm how the change will affect families, staff, and existing workflows, especially payment and communication behavior.

</Callout>

## How to open Site Settings

Use the Admin menu to reach the Site Settings screen.

<Steps>
  <Step title="Open the Admin menu" icon="settings" titleType="p">

    From the left sidebar, click **Admin**.

  </Step>

  <Step title="Go to Site Settings" icon="settings" titleType="p">

    In the Admin menu, click **Site Settings**. This opens the main Site Settings screen for your account.

  </Step>

  <Step title="Confirm you see the Site Settings tabs" icon="check-circle" titleType="p">

    At the top of the page, you should see a row of tabs such as **Modules**, **Basic**, **Payment Info**, **Mobile App**, and **Admin** (Admin appears only for superusers).

  </Step>
</Steps>

When the page loads, the **Modules** tab appears first. Other tabs load on demand when you click them.

## How saving and validation work

Site Settings use an autosave pattern instead of a single Save button.

- The page loads the **Modules** tab immediately, and other tabs load via background requests when you click them.
- Each input that participates in autosave has the `save-input` class and saves as soon as you change it (for example, toggling a checkbox, picking a value, or leaving a text field).
- The browser sends the new value to a save endpoint with the setting key and your account identifier.

<Callout kind="info">

Watch for inline success or error messages after you change a field. If a validation fails, Campium does not store the new value.

</Callout>

Key behaviors from the save logic:

- **Boolean toggles**: When you click a checkbox or toggle, Campium flips the stored value between `0`/`off` and `1`/`on`. The UI represents the current stored state.
- **`parent_contact_email` validation**: The **Contact Email Address** field on the Basic tab must pass Campium email verification (`isVerifiedEmail`). If verification fails, the save returns an error and the value does not change.
- **`group_field` side effect**: Changing the **Group Field** select on the Basic or Mobile App tab calls `GroupCustom::setCustomID` with the new value. This updates the underlying group field configuration in addition to storing the setting.
- **`surcharge` normalization**: When you change **Credit Card Surcharge**, Campium strips the percent symbol from the stored value. You can enter `3` or `3%`; the value saved for `surcharge` is numeric-like (for example, `3`).
- **Skipped keys**: The save endpoint ignores internal keys such as `online_id`, `key`, `PHPSESSID`, `username`, and any key starting with `_`. These are not stored as settings.

If you do not see the **Admin** tab, your user account is not a superuser; those settings are restricted.

## Tips for working safely with Site Settings

<Callout kind="tip">

- **Change one thing at a time**: Because settings auto-save, adjust a single field, confirm the behavior, then move on.
- **Document sensitive changes**: For payment processors and Admin tab options, note what changed, when, and why for your internal records.
- **Use test mode where possible**: For processors that support a TEST environment, validate configuration there before switching to LIVE.

</Callout>
